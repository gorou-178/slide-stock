# ADR-005: メモ機能の入力制約追加

## ステータス
Accepted

## コンテキスト

### 現状の実装

メモ機能（`PUT /api/stocks/:id/memo`）の入力バリデーションは以下の状態にある:

| 項目 | 現状 |
|------|------|
| 文字数上限 | 10,000文字 |
| 改行コード正規化 | なし |
| 前後トリム | 空文字チェック時のみ使用、DB保存はオリジナル値 |

### 課題

- 個人ツールのメモ用途として10,000文字は過大。200文字が適切。
- CRLF（Windows）、CR（旧Mac）などの改行コードが混在するとDB内のデータが不統一になる。
- 前後の空白がDB に保存されており、表示・検索で不整合を招く可能性がある。

## 決定

### 1. 文字数上限を200文字に変更

- `MEMO_MAX_LENGTH = 200`
- エラーメッセージ: 「メモは200文字以内で入力してください」
- フロントエンド: `maxlength="200"`、表示 `0 / 200`、警告ライン `count > 190`

### 2. 改行コードをLFに統一

- 正規化式: `memoText.replace(/\r\n/g, '\n').replace(/\r/g, '\n')`
- 適用タイミング: サーバー側、バリデーション前（trim後に実施）
- フロント側は任意（サーバー側で必ず正規化するため）

### 3. 入力値の前後トリム

- `const trimmed = memoText.trim()` として trim済み値で以降の処理をすべて統一
- DB保存も trim・正規化済み文字列を使用
- フロント側にも既存のtrim処理があるが（`stock-detail.astro:189`）、trim は冪等なので問題なし

## 実装設計

### サーバー側処理順序（`worker/handlers/memo.ts`）

```
1. trim()
2. 改行コード正規化（CRLF/CR → LF）
3. 空文字チェック（trim後の文字列で）
4. 長さチェック（trim・正規化後の文字列で）
5. DB保存（加工済み文字列で）
```

### 変更対象ファイル

| ファイル | 変更内容 |
|---------|---------|
| `worker/handlers/memo.ts` | `MEMO_MAX_LENGTH` を200に変更、trim・改行正規化追加、エラーメッセージ変更、DB保存変数を正規化済みに変更 |
| `worker/handlers/memo.test.ts` | M3（最大200文字）、M8（201文字超過）のテストデータ更新 |
| `src/components/MemoEditor.astro` | `maxlength="200"`、文字カウント表示 `0 / 200` |
| `src/pages/stock-detail.astro` | 文字カウント表示・警告ライン（190超）・エラーライン（200超）更新 |

## 却下した代替案

| 代替案 | 却下理由 |
|--------|---------|
| フロント側のみでバリデーション | サーバーサイド必須という方針に反する |
| trim後に長さチェック、保存はオリジナル値 | trim前後で長さが変わる可能性があり不整合 |
| 改行正規化をフロント側のみで実施 | サーバーサイド必須という方針に反する |
| 長さチェック → 改行正規化 → trim の順 | 正規化前の文字数でチェックされてしまい不整合 |
