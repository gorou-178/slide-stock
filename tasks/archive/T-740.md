# T-740 — [設計] ドキュメント統合: 5ドキュメント体制への再編

## 背景

docs/ に仕様書11本 + ADR 11本が散在しており、個人ツールとしては過剰。
5つのドメイン別ドキュメントに統合して見通しを改善する。

## 新構成（5ドキュメント）

### 1. `docs/frontend-spec.md` — フロントエンド
画面定義、画面フロー、デザインルール

**統合元:**
- `ui-spec.md` (1,151行) — 画面構成・コンポーネント・デザイントークン・レスポンシブ
- `landing-spec.md` (498行) — ランディングページ仕様

### 2. `docs/backend-spec.md` — バックエンド
API仕様、認証と認可、oEmbedやプロバイダの仕様

**統合元:**
- `stock-api-spec.md` (762行) — Stock CRUD API
- `memo-api-spec.md` (446行) — Memo API
- `auth-spec.md` (516行) — 認証フロー・エンドポイント定義・認可ミドルウェア
- `oembed-spec.md` (496行) — oEmbed メタデータ取得仕様
- `provider-spec.md` (484行) — プロバイダ検出・URL正規化

### 3. `docs/architecture-spec.md` — アーキテクチャ
インフラ層、CI/CD、cookieやセッション管理、セキュリティ対策

**統合元:**
- `architecture.md` (224行) — システム全体構成・デプロイ構成
- `auth-spec.md` から Cookie/セッション管理セクション（§5〜§9）を移動

**注意:** auth-spec.md は2つに分割される
- エンドポイント定義・認可ロジック → backend-spec.md
- Cookie属性・セッション管理・セキュリティ設計判断 → architecture-spec.md

### 4. `docs/data-model-spec.md` — データモデル
ER図、リレーション、マイグレーション

**統合元:**
- `database.md` (136行) — ER図・テーブル定義
- `plan-db-types.md` (180行) — DB型検討メモ（有効な内容のみ吸収、残りは廃止）

### 5. `docs/workflow.md` — 開発ワークフロー
タスク管理、テスト手法、コード管理

**統合元:**
- `task-workflow.md` (244行) — タスク運用ルール

## ADR の扱い

`docs/adr/` はそのまま維持。ADR は「いつ・なぜ・どう決めたか」の歴史記録であり、
仕様書の統合とは独立して存在する（CLAUDE.md の Spec/ADR/実装の関係ルールに従う）。

## 統合後に削除するファイル

- `docs/ui-spec.md`
- `docs/landing-spec.md`
- `docs/stock-api-spec.md`
- `docs/memo-api-spec.md`
- `docs/auth-spec.md`
- `docs/oembed-spec.md`
- `docs/provider-spec.md`
- `docs/architecture.md`
- `docs/database.md`
- `docs/plan-db-types.md`
- `docs/task-workflow.md`

## 統合方針

- 各仕様書の内容は原則そのまま移植（リライトしない）
- セクション番号は新ドキュメント内で振り直す
- ドキュメント間の相互参照（リンク）を新ファイル名に更新
- AGENTS.md・CLAUDE.md 内の docs/ 参照も更新

## 成果物

- 新5ドキュメント
- 旧11ドキュメントの削除
- 関連ファイルの参照更新
