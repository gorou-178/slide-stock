# T-754 — [設計] DBスキーマ型最適化設計

## 背景

`migrations/0001_init.sql` で全カラムが TEXT 型になっている。
また、Queue廃止（T-711）により `stocks.status` カラムが不要になった（常に 'ready' 固定）。

## 現状スキーマ

```sql
-- users
id TEXT PRIMARY KEY,
google_sub TEXT UNIQUE NOT NULL,
email TEXT NOT NULL,
name TEXT NOT NULL,
created_at TEXT NOT NULL DEFAULT (datetime('now'))

-- stocks
id TEXT PRIMARY KEY,
user_id TEXT NOT NULL,
original_url TEXT NOT NULL,
canonical_url TEXT,
provider TEXT,
title TEXT,
author_name TEXT,
thumbnail_url TEXT,
embed_url TEXT,
status TEXT NOT NULL DEFAULT 'pending',  -- ← 不要
created_at TEXT NOT NULL DEFAULT (datetime('now')),
updated_at TEXT NOT NULL DEFAULT (datetime('now'))

-- memos
id TEXT PRIMARY KEY,
stock_id TEXT NOT NULL,
user_id TEXT NOT NULL,
memo_text TEXT NOT NULL,
created_at TEXT NOT NULL DEFAULT (datetime('now')),
updated_at TEXT NOT NULL DEFAULT (datetime('now'))
```

## 設計すべき内容

### 1. status カラム削除

- Queue廃止により pending/failed ステータスは存在しない
- `handleCreateStock` で常に `'ready'` をINSERTしているが、参照箇所なし
- マイグレーションで DROP COLUMN（SQLite 3.35.0+ で対応）
- D1 の SQLite バージョンを確認

### 2. 型の見直し（SQLite の型親和性を考慮）

SQLite は動的型付けのため、TEXT で統一しても実害は少ない。ただし:

- `created_at` / `updated_at`: TEXT (ISO 8601) のまま維持が妥当（SQLite の datetime 関数と互換）
- `id` 系: TEXT のまま（UUID文字列）
- 結論: **TEXT 統一を維持し、status 削除のみ実施** が現実的か検討

### 3. マイグレーション戦略

- `migrations/0002_remove_status.sql` を作成
- 既存データへの影響を確認
- ロールバック手順
