# T-756 — [設計] UUID v7導入 + publicId検討

## 背景

現状、全IDが `crypto.randomUUID()` (UUID v4) で生成されている:

- `worker/handlers/auth.ts:192` — userId
- `worker/handlers/stocks.ts:128` — stockId
- `worker/handlers/memo.ts:86` — memoId

また、`/api/me` エンドポイントで userId が外部に露出している。

## 設計すべき内容

### 1. UUID v7 導入の判断

メリット:
- 時系列ソート可能（created_at なしでも順序がわかる）
- DB index のフラグメンテーション軽減（B-tree に優しい）
- 一意性は UUID v4 と同等

デメリット:
- `crypto.randomUUID()` は v4 のみ → 自前実装 or ライブラリ必要
- Cloudflare Workers 環境での互換性確認が必要

実装方式:
- ライブラリ: `uuidv7` npm パッケージ or 自前実装（小さいので自前が妥当か）
- Cloudflare Workers の `crypto.getRandomValues()` 利用可否

### 2. publicId の必要性検討

- 現在 userId が `/api/me` で返されている
- ユーザーに見せるIDが必要なケースはあるか？
  - 現状: 個人ツールで他ユーザーとの共有機能なし
  - 将来: スライド共有URL等で userId が露出する可能性
- 判断基準:
  - publicId が不要なら、内部IDをそのまま v7 に移行
  - publicId が必要なら、users テーブルに `public_id TEXT UNIQUE` を追加

### 3. 既存データの移行

- 既存の v4 UUID はそのまま維持（互換性あり）
- 新規生成分のみ v7 に変更
- マイグレーション不要（ID形式の混在は許容）
